Investigating Dynamic SQL

Published on March 9, 2026 by Perez

The belief that "Dynamic SQL is slow" is a persistent myth, but it oversimplifies how database engines handle execution. Performance depends on whether the engine can reuse an existing execution plan. If the engine receives the exact same query text repeatedly, it can use the cached plan regardless of how the string was generated.

Developers often misuse dynamic SQL by concatenating literal values (e.g., WHERE id = 123). This changes the query text every time, forcing the engine to "hard parse" and generate a new plan, creating significant overhead.

Example: Parameterized Dynamic SQL


-- GOOD: Parameterized query (Plan Reusable)
DECLARE @sql NVARCHAR(MAX);
SET @sql = N'SELECT * FROM Users WHERE ID = @ID';
EXEC sp_executesql @sql, N'@ID INT', @ID = 123;

By using bind variables, the database sees a consistent template, allowing for effective plan caching. This debunks the idea that dynamic SQL is inherently slow.

Hard Parse
The costly process where the database engine must generate a new execution plan from scratch.
Bind Variables
Placeholders in a SQL statement that represent values, allowing for plan reuse.
